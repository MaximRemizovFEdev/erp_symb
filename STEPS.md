# План разработки нативного Node.js ERP-приложения

## 1. Цель переработки

Перейти от архитектуры Directus + PostgreSQL к самостоятельному Node.js-приложению.

Основные решения:

- backend: Node.js;
- API: HTTP REST;
- хранение данных: JSON-файлы на диске;
- база данных на первом этапе не используется;
- frontend можно подключить отдельным SPA-приложением после стабилизации API;
- вся бизнес-логика находится в коде приложения, а не в настройках внешней CMS.

Главная задача первого этапа: получить работающую ERP-основу с заказами, клиентами, позициями, оплатами, производством, выдачей и ролями без зависимости от Directus.

## 2. Предлагаемый стек

- Node.js LTS;
- TypeScript;
- Express или Fastify для HTTP API;
- Zod для валидации входных данных;
- JWT или cookie-session для авторизации;
- JSON-файлы как persistent storage;
- Vitest или Node test runner для тестов;
- pino или winston для логирования.

Рекомендуемый вариант: Fastify + TypeScript + Zod, так как это даст строгие схемы данных, хорошую производительность и понятную структуру API.

## 4. JSON-хранилище

JSON-хранилище должно быть отдельным слоем, чтобы в будущем можно было заменить его на SQLite или PostgreSQL без переписывания бизнес-логики.

Требования к хранилищу:

- чтение коллекции из файла;
- запись коллекции в файл;
- атомарная запись через временный файл и rename;
- автоматическое создание файлов при первом запуске;
- блокировка конкурентной записи внутри процесса;
- резервная копия перед критичными изменениями;
- единый формат ошибок.

Формат файла коллекции:

```json
{
  "version": 1,
  "items": []
}
```

Для MVP допустимо хранить все данные в отдельных JSON-файлах. При росте объёма данных следующим шагом будет SQLite.

## 5. Модель данных

### Основные сущности

- `users` - пользователи системы и учетные данные;
- `employees` - сотрудники и менеджеры;
- `customers` - физические лица;
- `customerCompanies` - юридические лица;
- `customerCompanyLinks` - связь клиента и компании;
- `orders` - заказы;
- `orderItems` - позиции заказов;
- `orderPayments` - оплаты;
- `paymentAllocations` - распределение оплат по заказам;
- `contractors` - собственное производство и внешние подрядчики;
- `orderStatuses` - статусы заказов;
- `productionStatuses` - статусы производства;
- `officeStatuses` - статусы офиса;
- `auditLog` - журнал действий.

### Принцип связей

В JSON не хранить вложенные копии связанных объектов. Все связи хранить через `id`.

Пример:

```json
{
  "id": "order_001",
  "customerId": "customer_001",
  "companyId": "company_001",
  "managerEmployeeId": "employee_001"
}
```

API может возвращать расширенные DTO с подставленными клиентом, компанией, менеджером и позициями.

## 6. Роли и права

Реализовать роли на уровне backend middleware и сервисов.

Роли:

- `admin` - полный доступ;
- `owner` - полный рабочий доступ без системных настроек;
- `manager` - свои клиенты, компании, заказы, оплаты;
- `office` - выдача, статусы офиса, добавление оплат;
- `production` - только производственные представления своего направления.

Права проверять в двух местах:

- на уровне маршрутов API;
- на уровне выборки данных, чтобы пользователь не получил лишние записи.

## 7. Основные API-разделы

### Оплаты

## 8. Автоматизации

Реализовать сервисы:

- генерация номера заказа `SO-00001`;
- автоматическое назначение менеджера по текущему пользователю;
- пересчёт позиции после изменения количества, цены или себестоимости;
- пересчёт заказа после изменения позиции;
- пересчёт оплат после создания или удаления распределения;
- обновление баланса клиента и компании;
- автоматический перенос заказа в офисную выдачу;
- синхронизация статусов офиса между заказом и позициями;
- журналирование важных действий.

Все расчёты должны быть чистыми функциями с тестами.

## 9. Финансовые формулы

Для позиции:

```text
orderSum = quantity * pricePerUnit
unitCost = contractor1Cost + contractor2Cost
totalCost = unitCost * quantity
managerCommissionSum = orderSum * managerPercent / 100
taxSum = orderSum * taxPercent / 100
profitSum = orderSum - totalCost - managerCommissionSum - taxSum
marginPercent = profitSum / orderSum * 100
```

Для заказа:

```text
orderSum = sum(orderItems.orderSum)
itemsTotalCost = sum(orderItems.totalCost)
itemsManagerCommissionSum = sum(orderItems.managerCommissionSum)
itemsTaxSum = sum(orderItems.taxSum)
profitSum = sum(orderItems.profitSum)
paidAmount = sum(paymentAllocations.amount)
paymentDue = orderSum - paidAmount
officePaymentDue = paymentDue для office_pickup
marginPercent = profitSum / orderSum * 100
```

## 10. Этапы разработки

### Этап 1. Каркас приложения

- создать `package.json`;
- настроить TypeScript;
- настроить dev/start/build scripts;
- поднять HTTP-сервер;
- добавить health endpoint;
- добавить базовую обработку ошибок;
- добавить логирование.

Результат: приложение запускается локально и отвечает на `/health`.

### Этап 2. JSON storage layer

- создать `data/`;
- реализовать `JsonStore`;
- реализовать атомарную запись;
- реализовать repositories для коллекций;
- добавить seed начальных справочников;
- добавить тесты на чтение и запись.

Результат: приложение умеет стабильно читать и сохранять данные в JSON.

### Этап 3. Базовые справочники

- сотрудники;
- пользователи;
- контрагенты;
- статусы заказов;
- статусы производства;
- статусы офиса.

Результат: есть API для чтения и редактирования справочников.

### Этап 4. Авторизация и роли

- добавить пользователей;
- реализовать login/logout/me;
- добавить middleware авторизации;
- добавить проверку ролей;
- скрывать финансовые поля от производства и офиса.

Результат: разные роли видят разные данные.

### Этап 5. Клиенты и компании

- CRUD клиентов;
- CRUD компаний;
- связь клиент-компания;
- выбор компании только из компаний клиента;
- расчёт баланса клиента и компании.

Результат: менеджер может вести CRM-базу.

### Этап 6. Заказы и позиции

- создание заказа;
- автонумерация;
- автоматическое назначение менеджера;
- добавление позиций;
- редактирование позиций;
- пересчёт сумм;
- статусы заказа и офиса.

Результат: можно создать полноценный заказ с позициями и расчётами.

### Этап 7. Оплаты и распределения

- создать оплату независимо от заказа;
- распределить оплату между заказами;
- считать оплачено, остаток, переплату;
- хранить нераспределённый остаток;
- обновлять балансы.

Результат: оплаты работают отдельно от заказов, но корректно пересчитывают финансы.

### Этап 8. Производственные экраны

- API для собственного производства;
- API для шелкографии;
- универсальная фильтрация по `contractorId`;
- разрешить менять только статус производства и комментарий;
- скрыть цены, себестоимость, прибыль и оплаты.

Результат: производство получает только техническую часть заказа.

### Этап 9. Офисная выдача

- список заказов на выдачу;
- изменение статуса офиса заказа;
- изменение статуса офиса позиции;
- добавление оплаты из офиса;
- автоматическая синхронизация статусов.

Результат: офис может выдавать заказы без доступа к лишним данным.

### Этап 10. Frontend MVP

- экран входа;
- список заказов;
- карточка заказа;
- форма заказа;
- клиенты и компании;
- оплаты;
- производство;
- офисная выдача;
- базовая навигация по ролям.

Результат: системой можно пользоваться через браузер.

### Этап 11. Тестирование

- unit-тесты расчётов;
- тесты JSON-хранилища;
- тесты прав доступа;
- API-тесты основных сценариев;
- тест создания заказа;
- тест оплаты и распределения;
- тест производственного доступа;
- тест офисной выдачи.

Результат: критичная бизнес-логика защищена тестами.

### Этап 12. Подготовка к миграции на БД

JSON-хранилище подходит для MVP, но нужно заранее оставить возможность перейти на SQLite или PostgreSQL.

Для этого:

- бизнес-логика не должна читать JSON напрямую;
- все операции с данными идут через repositories;
- `id` не зависят от формата хранения;
- DTO отделены от storage models;
- расчёты не зависят от storage layer.

Результат: при необходимости можно заменить JSON storage на SQLite/PostgreSQL без полной переписки приложения.

## 11. MVP-границы

В первый релиз включить:

- авторизацию;
- роли;
- клиентов;
- компании;
- заказы;
- позиции;
- оплаты;
- распределение оплат;
- производство;
- офисную выдачу;
- расчёты;
- JSON-хранилище;
- базовый frontend.

Не включать в первый релиз:

- склад;
- документы;
- сложные уведомления;
- интеграции с телефонией;
- интеграции с бухгалтерией;
- импорт из внешних систем;
- сложную аналитику.

## 12. Риски

- JSON-файлы плохо подходят для одновременной записи несколькими процессами;
- при большом объёме данных поиск будет медленным;
- нужна аккуратная атомарная запись, чтобы не повредить данные;
- нужно заранее отделить storage layer от бизнес-логики;
- права доступа нельзя реализовывать только на frontend.

## 13. Критерии готовности MVP

- приложение запускается одной командой;
- данные сохраняются между перезапусками;
- заказ можно создать, отредактировать и закрыть;
- позиции заказа пересчитываются автоматически;
- оплаты распределяются между заказами;
- менеджер видит только свои заказы;
- производство не видит финансовые данные;
- офис может принять оплату и выдать заказ;
- основные сценарии покрыты тестами.
